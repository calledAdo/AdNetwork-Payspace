# Ad Network Backend API

This document describes the backend as it exists today in
`packages/payspace_backend/src`.

It answers three questions:

1. which endpoints exist
2. what each endpoint is responsible for
3. which internal helper functions make the route work

Related design draft:

- [SLOT_LIFECYCLE_PROPOSAL.md](./SLOT_LIFECYCLE_PROPOSAL.md)
  Proposed change for separating slot creation from snippet-install readiness.

## High-Level Role

The backend now has four main responsibilities:

- tracking and snippet serving
- publisher profile and slot onboarding
- campaign onboarding
- gateway/agent binding inspection and assistant orchestration

The backend is no longer the canonical source for buyer-facing placement
details. Seller slot details now originate in the publisher slot flow and are
meant to be synchronized to the seller agent.

## Tracking Routes

Defined in [src/routes/tracking.ts](./src/routes/tracking.ts).

### Core Data Models

- `MetricsRecord`
  Shared counter record for either a seller-side `snippet_id` or buyer-side
  `booking_id`.
- `BookingTrackingRegistration`
  Buyer-side tracked asset registration record.
- `SellerSlotDetails`
  Stable seller-side slot metadata keyed by `snippet_id`, written by the
  publisher slot flow.
- `SnippetContentRecord`
  Mutable rendered snippet content keyed by `snippet_id`.
- `TrackingContext`
  Runtime-resolved tracking target for one request.
- `SnippetContentPayload`
  Signed payload sent by the seller agent when updating snippet content.

### `GET /tracking/image`

Purpose:

- records an impression
- increments metrics for any `snippet_id`
- increments metrics for any `booking_id`
- redirects to the booking image if the request is booking-tracked
- otherwise returns a transparent PNG

Is this endpoint still used?

Yes.

It is used in two ways:

- seller-side snippet serving
  The snippet loader pings seller impression tracking through
  `sellerImpressionTrackingUrl`.
- buyer-side booking tracking
  `tracked_image_url` generated by `POST /tracking/bookings/register` points to
  this endpoint, and the endpoint redirects to the actual ad image.

Inputs:

- query `snippet_id`
- query `booking_id`
- legacy query `id`

Main helpers involved:

- `resolveTrackingContext()`
- `recordTrackingEvent()`
- `bumpMetrics()`

### `GET /tracking/click`

Purpose:

- records a click
- increments seller and/or buyer metrics
- redirects to the final destination URL
- or returns `204` when `ping=1`

Used by:

- seller-side click ping in the snippet loader
- buyer-side tracked click URL flow

Main helpers:

- `resolveTrackingContext()`
- `recordTrackingEvent()`

### `GET /tracking/stats/:id`

Purpose:

- returns metrics for either a `snippet_id` or `booking_id`

Response shape:

- `metric_id`
- `impressions`
- `clicks`
- `ctr`
- `geo_impressions`
- `geo_clicks`

Main helpers:

- `readMetrics()`

### `GET /tracking/snippet-seen`
### `POST /tracking/snippet-seen`

Purpose:

- records that the snippet loader has successfully initialized on a real page
- transitions the slot from `awaiting_install` to `publishing`
- sends `publish_slot` to the active seller agent once, on first seen

Used by:

- the snippet loader returned by `GET /tracking/snippet/loader.js`

### `POST /tracking/bookings/register`

Purpose:

- creates a buyer-side booking tracking record
- returns a `booking_id`
- returns `tracked_image_url`
- returns `tracked_click_url`

Used by:

- buyer agent before creative handoff

Stored record:

- `BookingTrackingRegistration`

Main helpers:

- `makeId()`
- `buildTrackedImageUrl()`
- `buildBookingClickTrackingUrl()`

### `GET /tracking/bookings/:id`

Purpose:

- inspect one buyer-side booking registration and its metrics

### `POST /tracking/snippet-content`

Purpose:

- seller-side signed update of the current renderable snippet content

Requirements:

- `snippet_id`
- monotonic `version`
- `signature`
- payload must verify against the active seller agent pubkey for the slot owner

Main helpers:

- `getSnippetContentPayload()`
- `applySignedSnippetContentUpdate()`
- `lookupSlotSignerPubkey()`
- `verifySnippetContentSignature()`

### `GET /tracking/snippet/loader.js`

Purpose:

- serves the self-contained snippet loader JS for a `snippet_id`

What it does:

- loads slot details
- loads current snippet content
- renders the slot
- pings seller impression tracking
- pings seller click tracking on click

Main helpers:

- `readSlotDetails()`
- `readSnippetContent()`
- `buildSnippetImpressionTrackingUrl()`
- `buildSnippetClickTrackingUrl()`

### `GET /tracking/slot-details/:id`

Purpose:

- inspect stable slot details and current snippet content for one `snippet_id`

### `GET /tracking/snippet/:id`

Purpose:

- alias-style inspection route for the same snippet-level view

### `DELETE /tracking/snippet/active`

Purpose:

- deprecated endpoint kept only to return a migration hint

## Publisher Routes

Defined in [src/routes/publishers.ts](./src/routes/publishers.ts).

### Data Models

- `SellerProfile`
  Human publisher profile plus lightweight seller-agent runtime metadata.
- `SellerSlotDetails`
  Full seller-side slot payload stored separately from the publisher profile.

### `POST /publishers`

Purpose:

- create a publisher profile

Input:

- `owner_pubkey`
- `site_url`
- optional `block_id`

Important:

- this does not spawn the seller agent
- it only creates the publisher profile

### `GET /publishers`

Purpose:

- list publisher profiles

### `GET /publishers/:ownerPubkey`

Purpose:

- fetch one publisher profile
- includes `slot_count`
- includes `agent_binding`

### `GET /publishers/:ownerPubkey/slots`

Purpose:

- list all seller slot detail records for one publisher

### `POST /publishers/:ownerPubkey/slots`

Purpose:

- create or update one seller slot
- seed inactive snippet content if needed
- ensure the seller agent exists
- send the slot payload to the seller agent as `register_slot`

This is the canonical slot creation route.

Input fields:

- `snippet_id` optional
- `page_url`
- `dimensions`
- `min_amount_per_1000`
- `ad_position`
- `publication_mode`
- `keyword_flags`
- `policy_text`
- `metadata`

If `snippet_id` is omitted, the backend generates one.

Main helpers:

- `createSellerSlotDetails()`
- `seedSnippetContent()`
- `ensureSellerAgent()`
- `syncSlotsToAgent()`

### `GET /publishers/:ownerPubkey/metrics`

Purpose:

- proxy seller agent metrics from the gateway

### `POST /publishers/:ownerPubkey/chat`

Purpose:

- send owner chat to the active seller agent

### `POST /publishers/:ownerPubkey/agent`

Purpose:

- manually respawn the seller agent
- resync all stored slots to the agent via `register_slot`

### `DELETE /publishers/:ownerPubkey`

Purpose:

- terminate seller agent if active
- delete the publisher profile

## Campaign Routes

Defined in [src/routes/campaigns.ts](./src/routes/campaigns.ts).

### Data Models

- `CampaignCore`
  Business payload known before agent spawn.
- `CampaignDetails`
  Outer runtime metadata and status.

### `POST /campaigns`

Purpose:

- create a campaign
- immediately attempt buyer-agent spawn

Stored shape:

- `campaign_core`
- `block_id`
- `agent_id`
- `agent_pubkey`
- `agent_status`
- `status`

### `GET /campaigns`

Purpose:

- list campaigns

Filters:

- `owner_pubkey`
- `status`

### `GET /campaigns/:id`

Purpose:

- fetch one campaign
- includes `agent_binding`

### `GET /campaigns/:id/metrics`

Purpose:

- proxy buyer-agent metrics from the gateway

### `PATCH /campaigns/:id`

Purpose:

- update mutable campaign business fields
- forward `update_settings` owner command to the buyer agent if active

### `POST /campaigns/:id/chat`

Purpose:

- send owner chat to the active buyer agent

### `POST /campaigns/:id/command`

Purpose:

- send an explicit owner command to the active buyer agent

### `POST /campaigns/:id/agent`

Purpose:

- respawn the buyer agent
- optionally allow block reselection

### `DELETE /campaigns/:id`

Purpose:

- terminate buyer agent if active
- delete campaign record

## Assistant Routes

Defined in [src/routes/assistant.ts](./src/routes/assistant.ts).

### Role

The assistant route is an onboarding/orchestration surface, not the operational
agent itself.

It has two modes:

- backend onboarding guidance
- routed owner chat to an active agent

### `POST /assistant/messages`

Purpose:

- continue an assistant thread
- decide whether to answer from backend or route to an active agent

Important request fields:

- `thread_id` optional
- `subject_type`
- `subject_id`
- `message`
- `route_mode` = `auto | backend | agent`

Seller workflow guidance now emphasizes:

1. create publisher profile
2. derive slot parameters
3. confirm slot
4. submit slot
5. allow first slot submission to spawn/sync seller agent

Campaign workflow guidance now emphasizes:

1. finish campaign targeting and creative
2. confirm campaign payload
3. submit campaign
4. let buyer agent spawn from `campaign_core`

### `GET /assistant/threads/:id`

Purpose:

- inspect one persisted assistant thread

## Internal Backend Functions

These are the main non-route functions worth knowing about.

### Orchestration layer

Defined in [src/orchestration.ts](./src/orchestration.ts).

- `listGatewayNodes()`
  Returns persisted gateway nodes or a single env-based fallback node.
- `selectGatewayNode(preferredBlockId?)`
  Picks the target gateway node for agent spawn.
- `spawnAgentOnGateway(node, body)`
  Calls `POST {gateway_url}/agents/spawn`.
- `terminateAgentOnGateway(binding)`
  Calls `DELETE {gateway_url}/agents/:agent_id`.
- `fetchAgentMetrics(binding)`
  Calls `GET {gateway_url}/agents/:agent_id/metrics`.
- `sendOwnerChat(binding, message)`
  Calls `POST {gateway_url}/agents/:agent_id/chat`.
- `sendOwnerCommand(binding, command, params)`
  Calls `POST {gateway_url}/agents/:agent_id/command`.
- `readAgentBinding(...)`
  Reads a subject binding from file storage.
- `writeAgentBinding(binding)`
  Persists a subject binding.
- `markBindingTerminated(...)`
  Marks a binding terminated without deleting history.

### Store layer

Defined in [src/store.ts](./src/store.ts).

- `readRecord(collection, id)`
  Read one JSON record.
- `writeRecord(collection, id, data)`
  Atomically write one JSON record.
- `deleteRecord(collection, id)`
  Delete one record.
- `listRecords(collection)`
  List all records in a collection.
- `updateRecord(collection, id, updater)`
  Atomic read-transform-write helper.

## Current Canonical Write Paths

These are the intended mutation entry points now:

- publisher profile creation:
  `POST /publishers`
- seller slot creation:
  `POST /publishers/:ownerPubkey/slots`
- campaign creation:
  `POST /campaigns`
- booking tracking creation:
  `POST /tracking/bookings/register`
- snippet content update:
  `POST /tracking/snippet-content`

Slot creation should not go through `/tracking/*`.
